How Agentic AI Borrows From Human Work and Learning

Agentic AI is borrowing from two places: work practices that make outputs reliable, and feedback loops that help systems improve.

The Borrowing Has Moved Up the Stack

The idea that AI borrows from human life is not new. Neural networks are the obvious historical example: a computing architecture inspired, however loosely, by the way biological neurons connect and fire.

What feels different about the current wave of agentic AI is where the borrowing is happening. We are no longer only borrowing from biology at the level of model architecture. We are borrowing from the way humans organise work, pass judgement, learn from mistakes, ask clarifying questions, and improve through feedback.

That matters because the patterns emerging around agents are not a grab-bag of clever tricks. They are human habits that survived real work, translated into product and system design.

The First Source: Human Work Practices

Software engineering is a useful place to start because it has spent decades learning how to make imperfect human work safer without pretending humans are perfect. We use tests to make expectations executable. We use code review to add another layer of judgement. We use CI to catch regressions before they reach production. We use approval gates because some changes deserve friction.

Agentic systems are beginning to inherit those habits. Anthropic's public writing on multi-agent research systems describes a lead agent coordinating specialist subagents. Its documentation on evaluation workflows treats quality as something to test against examples and rubrics, not something to vibe-check after the fact. Their engineering guide to evals makes the same practical point: evaluation needs to be designed, measured, and iterated like the rest of the system. That is not alien to software engineering. It is recognisably the same family of ideas as review, testing, and acceptance criteria.

Orchestration is another borrowed pattern. A lead agent that holds the goal, decomposes the work, delegates to specialists, and integrates their findings is doing something recognisably human: holding the bigger picture so the people — or agents — closer to the task can focus on their slice. Multi-agent architectures encode a coordination habit humans have been refining for a long time.

The important move is the separation between doing and judging. A model can produce an answer, but another process can decide whether that answer actually satisfied the task. The reviewer may be a human, an automated eval, another agent, or some combination of all three. Either way, the system gets stronger when judgement is explicit instead of hidden inside a single generation step.

This is where a lot of AI discourse is still too prompt-centric. Better prompts help, but mature software teams do not rely on one perfect instruction to protect production. They build systems around the work: checks, reviewers, tests, observability, rollback paths, and norms about what good looks like. Matt Pocock makes a similar point in Software Fundamentals Matter More Than Ever: those fundamentals still apply, even more so when agents are driving the work.

The Second Source: Human Cognition

The second source is less about professional practice and more about cognition itself. Some emerging agent patterns look less like software process and more like attempts to copy how people learn.

Take the idea of agents reviewing their own traces after the fact, sometimes described with language like dreaming or consolidation. For me, the interesting part is not the branding. It is the mechanism: experience is not only used in the moment; it is replayed, compressed, and turned into better future behaviour. That is close to how sleep is often described in memory research, including Diekelmann and Born's review of sleep and memory consolidation. The pattern comes from cognitive science more than from software process.

Dialogue is another example. Matt Pocock's Grill Me skill for Claude Code is a small but sharp pattern: before the agent acts, it pushes back with clarifying questions until the user's intent is clearer. That is not just prompt decoration. It is a way of importing a human collaboration habit into the agent loop. People build shared understanding by explaining, questioning, restating, and correcting each other. Agents need versions of that too.

Two Sources, One Loop

These two sources work best together. Work practices make agentic systems safer: review, testing, CI, and approval gates protect quality because they make judgement explicit. Cognitive mechanisms help the system improve: dialogue, reflection, and consolidation turn feedback into better future behaviour.

That combination matters because AI makes output cheap. It can generate ten drafts, five plans, or a dozen plausible next actions without telling us which one is true, useful, safe, or worth committing to. The scarce skill becomes defining what good means, making that definition inspectable, and building loops that improve the next attempt.

The core loop is simple: generate → evaluate → revise → approve → learn. Software engineering gives us mature patterns for the evaluate and approve parts. Human cognition gives us patterns for the revise and learn parts.

What Comes Next

If this is the right frame, I think we should expect more agent designs to borrow from human collaboration. Two quick candidates stand out.

The first is social learning: agents improving by observing other agents' successes and failures, especially inside bounded domains where the work can be replayed and evaluated. The second is the post-mortem: failed agent runs should leave behind a useful diagnosis, not just an error state.

The stronger near-term example, though, is the shared working surface before commit. Coding agents already show the value of diff-first workflows: the agent proposes a change, the human reviews it, and persistence is gated by approval. That pattern is partly about quality, but it is also about responsibility. The human can see what changed before it becomes real.

I do not think that pattern has been generalised widely enough across normal SaaS yet, but it should be. It points at one of the hardest product questions in an agentic world: when an AI helps produce an output, how do we keep responsibility shared, legible, and reviewable?

This is not just theoretical for me. In a recent prototype I demoed, the AI chat and workflow inputs sat side by side. As the conversation unfolded, the agent naturally populated the form fields for the user to review, correct, and approve in context. The user was not chatting beside the product, then copying the result back into the workflow. The conversation and the business object shared the same surface.

That is a direct example of a coding-tool lesson moving into broader knowledge work. The agent can propose, draft, and structure the output, but the user stays close enough to steer, inspect, correct, and approve it. The product is not only generating content; it is designing the collaboration contract around that content.

That is the next step I expect to see more often: applying diff-and-approve thinking to business objects like development plans, invoices, CRM updates, reports, support replies, policy drafts, or performance notes. The agent drafts into the same surface the human uses, but the real commit still waits for review.

These examples are not exhaustive. The useful move is learning to recognise which human patterns have earned their place, then asking what they become when built directly into a product or agent loop.

The Patterns Are the Point

None of this means agentic AI should imitate humans for its own sake. Some human patterns are inefficient, biased, political, or irrelevant to machines. The useful move is selectivity, not anthropomorphism.

The best teams will ask which human practices survived because they solve real coordination, quality, and learning problems. They will ask which cognitive mechanisms are worth approximating because they help the system get better over time. Then they will turn the answers into explicit architecture: evals, rubrics, review surfaces, memory, post-mortems, and approval gates.

That is the useful shift. Better agents will not only do more work. They will be surrounded by the same kinds of feedback, judgement, and learning loops that make human work reliable.

The next wave of agentic AI will not only be built from better models. It will be built from better borrowed patterns.
